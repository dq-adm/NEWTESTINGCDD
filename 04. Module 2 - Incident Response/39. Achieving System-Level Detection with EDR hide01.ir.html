Endpoint System-level Detection

Almost all cyber attacks involve executing malicious code on an endpoint to gain access or maintain it. System-level detection is monitoring and collecting activity data from an endpoint to analyze and identify threat patterns. The most straightforward application of system-level detection is an AntiVirus detecting malware while being written to the disk or during execution.

Right now, EDR agents are the primary tool security analysts utilize to achieve system-level detection. Irrespective of the solution, system-level detection works by collecting and analyzing a bunch of activities, such as:
  • Processes activities: Any action performed on the endpoint must have an executing process. Therefore, process logs are the most critical component in system-level detection because they can often tell the story of how the malware got into the endpoint, what it did, and in which order.
  • Disk activities: attackers may create, change or delete files.
  • Windows Registry activities: registry is like the Windows database, and most attacks leave decent trails in the Windows registry, such as setting up a service to start every time the host starts up (i.e., after a reboot).
  • Account management activities: user creation, deletion, and changes such as password reset and account lockout.
  • Account login activities: user login and logout attempts.
  • Log management: log deletion could indicate attackers trying to remove their traces.
  • Powershell activities: attackers utilize PowerShell heavily to automate tasks and control the compromised machine. Monitoring executed commands provide visibility into attacker moves.


Windows stores all of its logs in a centralized place called Windows Event Log. A typical task for detection engineers is determining what logs to enable or disable. Some logs may eat up storage and present no security value, while others are very important such as process creation logs, which need to be manually enabled. The way system-level detection works often follow these steps:

  1. Configure auditing policy, using GPO to determine what logs should be enabled on the enterprise endpoints.
  2. Forward generated logs (telemetry) from endpoints to a centralized place (SIEM) for parsing and monitoring.
  3. Create SIEM alerts for specific use cases.


Unfortunately, the built-in Windows logging capability has many limitations, either by not having the ability to log certain events or the inability to specify logging criteria (e.g., creating exclusions). For example, detection-wise, we may be interested in logging DNS queries made by powershell.exe as it may indicate a compromised machine communicating with the C2 server. The built-in logging cannot:

  • Log DNS queries on endpoints
  • Specify logging criteria; Do not log all DNS queries, only ones created by powershell.exe.


Fortunately, third-party tools can extend the built-in logging capability and provide granular customization, such as Sysmon, which can log DNS queries and allow you to create exclusions. Sysmon is a free tool written by Mark Russinovich, Microsoft Azure CTO.

Now comes the critical question, how do I know what logs to collect and what exclusions to include? One of the best things about the CyberSecurity field is the strong, collaborative community that stands behind it. Sysmon configurations can be an excellent start for decent system-level detections, such as the popular SwiftOnSecurity Sysmon configuration file. Do not worry about understanding everything inside this configuration file right now; we have a practical lab explaining how to get started with Sysmon in detail.

One thing worth mentioning about Sysmon is that it has always been known and used as a telemetry generation tool only until recently when it introduced the ability to react and block malicious files, a basic prevention mechanism.

Besides Sysmon, the free tool, there are commercial solutions that provide similar capabilities and extra features, with Microsoft Defender being the most popular. Defender is a commercial detection/response solution that includes an AV engine and comes in with a pre-determined list of detection logic, unlike Sysmon, where you will create this list from scratch.

Both tools are Microsoft tools and outstanding options for telemetry generations, yet, the selection decision should consider many factors such as team maturity, budget, and available resources.

Another significant difference is that attackers often perceive Microsoft Defender as an annoying AntiVirus they should disable on the target endpoint to be able to control it without restrictions, unlike Sysmon, which does not have an AV engine. That resulted in Sysmon providing better visibility during attacks because attackers often overlook it.
