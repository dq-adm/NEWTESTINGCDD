Memory Forensics > [T5]: Parsing MFT entries in Memory

Now that we have seen how MFT contains a trove of important information, the next question would be; how to inspect MFT content? The MFT parser plugin extracts and analyzes crucial metadata from Master File Table entries, providing valuable insights into files and directories on an NTFS file system. 

MFT ENTRY structure

The plugin extracts MFT entries in memory by searching for the "FILE" and "BAAD" strings, which are identifiers for MFT records in memory. The Master File Table (MFT) has multiple fields for each record, and the plugin can identify and parse the content of the following fields:

  • $FILE_NAME ($FN): stores the name of the file or directory, along with additional metadata like timestamps (creation, modification, and access times), allocated size, and parent directory reference. This attribute assists in reconstructing the directory structure and analyzing file activities during investigations.
     
  • $STANDARD_INFORMATION ($SI): contains essential metadata about a file or directory, such as file ownership, permissions, and more detailed timestamps (creation, modification, and access times).
     
  • $DATA: responsible for storing the actual content of a file or directory in an NTFS file system. It can either store the data directly within the MFT entry (for small files ~600 bytes) as resident data or point to the data's location on the disk as non-resident data. This is useful in incident response as you can inspect the actual content of tiny suspicious files or scripts that an attacker may have used.

Let's see how the mftparser plugin works:

mftparser - zoom in

  1. Volatility command line executing mftparser plugin.
  2. The first section of output contains:
    1. The physical offset of the MFT record in memory.
       
    2. Attribute: This field in MFT records contains information about the type of object, whether it is a file or directory, and whether the object has been deleted. The "In use" attribute indicates that the file or directory is currently in use and has not been deleted.
       
    3. Record number: This field contains the sequence number of the record within the Master File Table. It identifies the order of the record inside the MFT.
       
    4. Link count: The number of hard links to the file system file. Hard links are multiple file names that point to the same physical file on a disk. This attribute can be useful in tracking down malicious files that may have been created with a different name but still point to the same underlying content. By examining the link count, an analyst can determine if a file has additional names, which can aid in identifying any potential duplication or deception. (https://sabercomlogica.com/en/ntfs-mft-metadata-files/)
       
  3. $STANDARD_INFORMATION: contains the timestamps when the file was created, last modified, last accessed, and last time the MFT record changed and the file's type. $SI is useful for building a timeline of file activity and determining when a file was last accessed or modified. If it is given a suspicious type of "Read-only", "Hidden", or "SYSTEM," these types are given by attackers to the malicious files they drop on the machine to evade detection and bypass security measures. Attackers may give their file the "System" attribute to protect it from being deleted.

    Here is a list of entries you may find under the type section:
     
    1. READ ONLY: The file can only be read and not modified.
    2. HIDDEN: The file is hidden from normal directory listings.
    3. SYSTEM: The file is a part of the operating system and is required to function properly.
    4. ARCHIVE: The file has been modified since the last backup.
    5. DEVICE: The file is a device driver.
    6. NORMAL: The file is a regular file and has no special attributes.
    7. TEMPORARY: The file is temporary and may be deleted by the system when no longer needed.
    8. SPARSE FILE: The file has been allocated in a way that optimizes the use of disk space.
    9. REPARSE POINT: The file is a symbolic link or junction point.
    10. COMPRESSED: The file has been compressed to save disk space.
    11. OFFLINE: The file is not currently available on the system and may need to be retrieved from an external source.
    12. NOT CONTENT INDEXED: The Windows Search service does not index the file.
    13. ENCRYPTED: The file has been encrypted to protect its contents.
       
  4. $FILE_NAME ($FN): contains the file name and timestamps. The $FN attribute is often used to identify files and directories and their properties.

    As you may notice, The $STANDARD_INFORMATION and $FILE_NAME attributes in Windows file systems contain different timestamps because they are populated differently. User-level processes update $SI timestamps, while $FILE_NAME timestamps are updated by the system kernel, making them less susceptible to anti-forensic activities. SANS Institute has published a comprehensive poster explaining when each attribute will be updated.
