(4) Process privileges - GETSIDS plugin

The GETSIDS Volatility plugin extracts and display the security identifiers (SIDs) of all user accounts that have started a process. When a user starts a process, the process inherits the user's SID through the Process Environment Block (PEB). The GETSIDS plugin scans the PEBs of all running processes in a memory dump and identifies the SIDs of the users who started them, providing a way to map processes to the users who started them and identify any unexpected or unauthorized processes running under a particular user account.

Security Identifiers (SIDs) are unique alphanumeric codes assigned to every user, group, and computer in a Windows operating system. SIDs are used to control access to resources and to identify the owner of a particular process. Some examples of popular SIDs are: 

  • S-1-5-32-544: Administrators group
  • S-1-5-32-545: Users Group
  • S-1-5-32-546: Guests group
  • S-1-1-0: Everyone group
  • S-1-5-21-domain-512: Domain Admins group for the domain
  • S-1-5-18: Local System account
  • S-1-5-19: Local Service account
  • S-1-5-20: Network Service account
  • S-1-5-21-domain-500: Administrator account for the local machine
  • S-1-5-21-domain-1001: User account in the domain

A comprehensive list of well-known Windows SIDs can be found on the understand-security-identifiers page. To see SIDs in action, execute 'whoami /all' in your Windows command line, and you will see the logged-in user account SID and the SIDs of the groups the user is a member of.

 

By analyzing the SIDs associated with a process, you can determine whether the process is associated with a legitimate user account or an attacker who may have gained unauthorized access to the system. 

In addition, system processes (e.g., system.exe, lsass.exe, and svchost.exe) are expected to run with system accounts' privileges (i.e., SID). Attackers may create a malicious process and name it after well-known system processes to avoid detection. Thus, seeing a system-like process started by a standard user account (usually a compromised account) is abnormal. 

By mapping each process to the user who started it, GETSIDS not only helps in detecting abnormalities, but it can also help trace back how the malware found its way to the system (i.e., the compromised account).

Let's continue with the same example and run the 'getsids' plugin against the 'CCDrootkit' process.

  1. The physical address of the investigated process (CCDrootkit). Note that you can also specify the process using its PID.
  2. The SID and name of the user account that started the process.
  3. The group membership of the user who started the process (i.e., IT_Manager). As shown, it's a local admin and domain admins group member. 
  4. The Mandatory level refers to the security level assigned to the process. It is used to enforce system security policies, such as mandatory access control, which restricts access to resources based on their security level. The security levels include Low, Medium, High (Administrators), and System, with higher levels having greater access to resources.

    Is not knowing the user account SID sufficient? The answer is no. Even if you figured out the account that started the process, you must know its privilege. For example, a local administrator can start a program/executable/process using administrator or system privileges. Check out this article for examples of the difference between administrator and system privileges.
