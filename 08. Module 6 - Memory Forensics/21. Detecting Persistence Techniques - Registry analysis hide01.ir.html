Memory Forensics > [T4]: Detecting Persistence Techniques: Registry analysis

(1) Volatility printkey plugin

The printkey volatility plugin can assist you in detecting persistence techniques by analyzing persistence-associated registry keys and values. 

Its primary use is to list the registry keys and values under a specific registry key. By reviewing relevant keys, you can identify if an attacker has established persistence on a system and determine the methods used to do so. This data can then be used to develop remediation strategies to prevent repeated compromise.

 

printkey - zoom in

Let's look at the PRINTKEY output parsing the content of the popular registry key  ''software\microsoft\windows\currentversion\run" in a clean state.

Volatility command consists of multiple parts. The -g 0xf803788a44d8 is the offset of kdCopyDataBlock, feeding volatility the address to skip scanning the whole memory dump. And the printkey -K "software\microsoft\windows\currentversion\run" is the inspected registry key.

Now, let's break down the plugin output:
  • 1. Hive path: hive path determine if the configuration stored is specific to a particular user or applies to the entire machine. 
    • Hives in "\SystemRoot" indicate that the configuration applies to all users.
    • Hives in the current user's profile directory (e.g., "C:\Users\username\NTUSER.DAT") indicate that the configuration only applies to that user.
  • 2. Key name and type: "(S)" means that the key is a stable registry key permanently stored on the hard drive. If the key is marked as "(V)," it is a temporary/volatile registry key that exists only in memory. Volatile keys store system settings that frequently change, while stable keys are used for configuration data that needs to persist between system restarts. It makes sense that attackers will be interested in stable keys as their objective is to achieve persistence.
  • 3. Last updated: the date and time when the key was last modified. This information can help establish a timeline of events during the incident.
  • 4. Subkeys: contain a list of subkeys that exist under the inspected (parent) key, if any, along with their type, stable (S) or volatile (V).
  • The "Values" section displays key content/values. It's broken down into: 
    • 5. Value name: usually the application name.
    • 6. Value data: the key content. Usually, an executable or a command line.
    • 7. Value type: Stable or volatile, similar to registry keys.
    • 8. Value data type: the data type. For a complete list of possible types, please check out Registry value types by Microsoft. Knowing the data type will help you figure out the proper way to decode it.
