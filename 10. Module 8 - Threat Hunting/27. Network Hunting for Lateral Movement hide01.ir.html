Threat Hunting > [T4]: Network Hunting for Lateral Movement

Lateral movement refers to the techniques and methods used to move through a network or system from one compromised host to another to gain access to sensitive data or systems. In a Windows enterprise domain, lateral movement often involves leveraging compromised credentials to move laterally from one host to another within the network.

 

Attackers may use a variety of techniques to achieve lateral movement, including:

  1. Pass-the-hash: Using harvested password hashes to authenticate to other systems in the network.

  2. Pass-the-ticket: Using Kerberos ticket-granting tickets (TGTs) to authenticate to other systems in the network.

  3. Remote code execution: Using an exploit to gain control of a system and then using that system to move laterally to other systems in the network.

  4. Service impersonation: Using legitimate service accounts to move laterally through the network.

 

Once an attacker has gained access to a system, they will typically attempt to escalate their privileges to gain access to more sensitive data or systems. This often involves the use of tools such as Mimikatz or PowerSploit to harvest credentials or to manipulate security tokens.

 

Effective defense against lateral movement requires a comprehensive approach, including monitoring and logging network traffic, detecting anomalous activity, and managing user accounts and credentials. Additionally, organizations should implement security controls such as network segmentation, multifactor authentication, and endpoint protection to reduce the risk of lateral movement in case of a compromise.

 

When using Mimikatz to perform pass-the-hash attacks, an Event ID 4624 (Logon/Sign-in) is usually generated on the compromised endpoint. The exact artifacts recorded in this event may vary depending on the version of Windows and security auditing settings in place, but some common information that may be included are:

  • EventID: This is the numerical identifier for the event, which should be set to 4624.

  • LogonProcessName: This field indicates the name of the process that was responsible for the logon. For Mimikatz pass-the-hash attacks, the logon process name is usually set to "advapi" or "seclogo".

  • Logon type: This field indicates the type of logon that was performed. For pass-the-hash attacks, the logon type is usually set to "(Logon Type 9) NewCredentials"

  • Authentication package: This field indicates the name of the authentication package that was used to perform the logon. For pass-the-hash attacks, the authentication package is usually set to "Negotiate."

 

First, let us check for authentication with a logon type set to 9, which is NewCredentials and AuthenticationPackage set to Negotiate, along with a logon processname set either to advapi or seclogo

KQL Query: event.type: "start" AND event.outcome: "success" AND (event.category : "authentication" and winlog.logon.type: "NewCredentials" AND winlog.event_data.AuthenticationPackageName: "Negotiate" AND winlog.event_data.LogonProcessName: ("advapi" OR "seclogo") )

 

As you can see, there is a successful match. Now let us check the surrounding events to see if any credential dumping happened before that event on the same endpoint.

And to do that, we need to look for events matching the following logic, a process that accessed lsass, and here is the equivalent KQL query.

event.category: "process" AND event.type: "access" AND winlog.event_data.TargetImage: "C:\Windows\system32\lsass.exe"
