Disk Forensics > [T6]: Object Access Logging: Monitoring File/Folder Activity

(1) Security.evtx

By default, Windows does not log File/Folder (object) access. To do so, first, you will have to enable object auditing on the system (where the resources you want to monitor exist) using the security policy. You can do that through the local security policy on a standalone machine or GPO in a domain environment.

The second step is configuring auditing on the specific object (file/folder) you want to monitor. You may wonder, what is the benefit of the first step? The answer is simple. The first step is to activate the logging component without determining what objects to audit. Windows cannot audit all objects on a system because the system will break (too many events). Therefore, you have to select and configure what objects to audit manually.

 

To try this out on your local machine:

  1. First, run this command to activate the logging component: "auditpol /set /category:"Object Access" /success:enable /failure:enable" from a privileged command-line.
  2. Next, identify the files you wish to monitor. In our video example below, I created a simple dummy text file for demonstration purposes.
  3. Ensure the file's permissions are set to restrict access to your current user.
  4. Lastly, enable auditing for any failed attempts to interact with the file.

 

There are four primary event IDs related to object access in "Security.evtx":

  • Event ID 4656: is the first event in the "Accessing an object" chain. It will be logged once an access attempt is made, regardless of whether it was successful or failed.
  • Event ID 4660: This event is logged when an object is deleted.
  • Event ID 4663: This event is logged when the access attempt (ID 4656) is successful. In other words, the user successfully accessed the object.
  • Event ID 4658: This event is logged when closing the handle to the object. In other words, when access ends.

4656 event - (zoom in)

 

  1. A handle to an object was requested (4656) event.
  2. The subject section contains information about the account that accessed the file. You can see that the user who performed this action is "IEUser".
  3. Type of the object, a file, or a folder.
  4. The object path
  5. The "Handle ID" is a unique identifier assigned to each object (file/folder). You can use it to correlate all actions performed on that object. For example, if a user accessed a file and deleted it,  A 4656 and 4660 events will be logged for the same Handle.
  6. Process information: which process (i.e., application) accessed the file?

 

Event ID 4656 indicates that a handle to an object was opened or closed. In other words, an attempt to access an object, such as a file, folder, or registry key, and whether the access attempt was successful or unsuccessful.

Following the occurrence of Event ID 4656, other events such as Event ID 4659, Event ID 4663, and Event ID 4660 may also be generated, providing further details about the specific actions taken on the object and additional information about the user who performed the action. For example, 4656 followed by 4660 means the user opened a handle to the object and then deleted it.
