Disk Forensics > [T9]: Windows Services: Configuration, Timeline, and Events

(1) Windows Services

Threat actors often use Windows services to persist on the target system and ensure that they won't lose access. The configuration of Windows services is stored in the System Registry hive at C:\Windows\System32\config\SYSTEM under the "CurrentControlSet\Services" key.

One thing to note is that you may see other variants of "ControlSet" keys under the SYSTEM hive, such as "ControlSet002", "ControlSet003", etc. The operating system uses one of them as an active configuration profile, while the rest serve as backups. Those backups are used to load the "Last Known Good Configuration" boot option. To determine the active/loaded "ControlSet," check the "HKLM\SYSTEM\Select" key, and you will see the loaded "ControlSet" under the key value "current".

 

Using the 'service' plugin in Registry Explorer to parse services information

To create a timeline of when a service was installed and when it was started, you can check the Security and System event logs for certain event IDs. These event IDs track the lifetime of service such as:

  • 4697 (Security.evtx): This event is generated when a service is installed, and it contains information about the executable path, service name, and the account that installed the service. For this event to be generated, you need to enable the "Security System Extension" audit policy using the command "AuditPol.exe /set /subcategory:"Security System Extension" /failure:enable /success:enable".
  • 7034 (System.evtx): This event is generated when a service crashes (possibly due to process injection).
  • 7035 (System.evtx): This event is generated when the OS sends a start/stop control signal to the service.
  • 7036 (System.evtx): This event is generated when the service is actually started/stopped.
  • 7040 (System.evtx): This event is generated when the start type of a service is changed (e.g., auto, manual, automatic-delayed, and disabled), which may indicate persistence. 
  • 7045 (System.evtx): This event is similar to the 4697 events but does not include information about the account that installed the service.

By examining these events, you can build a timeline of when a service was installed, stopped, started, or changed and gain insights into its behavior on the system.
